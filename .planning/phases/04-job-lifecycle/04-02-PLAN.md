<task type="auto">
  <name>Task 2: JobService — state machine, CRUD, and scheduling integration</name>
  <files>backend/app/features/jobs/service.py</files>
  <action>
Create `backend/app/features/jobs/service.py` with the state machine constants and `JobService(TenantScopedService[Job])`.

**State machine constants** (module-level):

- `JobStatus(StrEnum)` — import from schemas or define here if needed
- `ALLOWED_TRANSITIONS: dict[tuple[str, str], frozenset[str]]` — maps (current_status, role) to allowed next statuses. Per CONTEXT.md locked decisions:
  - Admin: all transitions (forward and backward)
  - Contractor: Scheduled->In Progress, In Progress->Complete (own jobs only)
  - Client: view only (no transitions — empty frozensets)
  - Cancelled is terminal (no transitions out for any role)
- `BACKWARD_TRANSITIONS: set[tuple[str, str]]` — pairs like (scheduled, quote), (in_progress, scheduled), (complete, in_progress), (invoiced, complete)
- `is_backward(current, next_status)` helper function
- `InvalidTransitionError` exception class with from_status, to_status, role attributes

**JobService class methods:**

1. **create_job(data: JobCreate, user_id, company_id)** — Create Job from schema data. Set initial status (defaults to 'quote' but company-assigned jobs can start at any stage per user decision). Append initial status_history entry: `{status, timestamp, user_id, reason: "Job created"}`. Return created Job. Use `db.flush()` to get the generated ID (per CLAUDE.md — no db.commit()).

2. **transition_status(job_id, new_status, role, user_id, reason, expected_version)** — The core state machine method:
   a. Fetch job via repository.get_by_id
   b. 404 if not found
   c. Check expected_version matches job.version (409 Conflict if mismatch — per Pitfall 2, prevents offline/online race)
   d. Look up (job.status, role) in ALLOWED_TRANSITIONS
   e. Raise InvalidTransitionError if new_status not in allowed set
   f. If is_backward and no reason: raise 422 "A reason is required for backward transitions"
   g. If contractor role: verify contractor_id matches user_id (own jobs only)
   h. Append status_history entry using list replacement (NOT in-place append — per Pitfall 3): `job.status_history = [*job.status_history, new_entry]`
   i. Update job.status
   j. Increment job.version
   k. If new_status == "cancelled": call repository.cancel_job_bookings(job_id) for bulk booking soft-delete
   l. If backward transition frees scheduling: call repository.cancel_job_bookings(job_id) (e.g., Complete->In Progress means existing bookings need to be freed for re-scheduling)
   m. `db.flush()` + `db.refresh(job)` + return job

3. **update_job(job_id, data: JobUpdate)** — Partial update for non-lifecycle fields. Increment version. Return updated Job.

4. **search_jobs(query, filters)** — Delegate to repository.search_jobs.

5. **get_job(job_id)** — Delegate to repository.get_by_id.

6. **list_jobs(filters)** — Delegate to repository.list_jobs.

7. **get_contractor_jobs(contractor_id)** — Delegate to repository.get_jobs_for_contractor.

8. **soft_delete_job(job_id, user_id)** — Set deleted_at, also cancel_job_bookings. Add "cancelled" status_history entry if not already cancelled.

Do NOT import SchedulingService directly into JobService at this stage. The scheduling integration (calling book_slot when transitioning Quote->Scheduled) will be wired at the router level in Plan 04, where both services are available via dependency injection. This keeps JobService focused on the state machine and avoids circular dependency between features.
