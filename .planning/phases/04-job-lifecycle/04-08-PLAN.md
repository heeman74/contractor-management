Setup: Create helper functions to create a test job and transition it to desired status.

Tests:
1. **test_create_job** — POST /api/v1/jobs/ with valid data. Assert 201, response contains all fields, status='quote', status_history has initial entry.
2. **test_create_job_with_initial_status** — POST with status='scheduled' (company-assigned skip-ahead). Assert status='scheduled'.

6. **test_transition_backward_requires_reason** — PATCH with backward transition (scheduled->quote) without reason. Assert 422.
7. **test_transition_backward_with_reason** — PATCH with backward transition + reason. Assert 200.
8. **test_transition_invalid_rejected** — PATCH contractor trying quote->scheduled. Assert 422.
9. **test_transition_version_mismatch** — Create job, transition once (version becomes 2), then try transition with version=1. Assert 409.
10. **test_cancel_job_frees_bookings** — Create job, create associated booking (via scheduling endpoint), cancel job. Assert booking.deleted_at is set.
11. **test_update_job** — PATCH /api/v1/jobs/{id} with updated description. Assert 200 with new values.
12. **test_soft_delete_job** — DELETE /api/v1/jobs/{id}. Assert 204. GET returns 404.
13. **test_search_jobs** — Create jobs with known descriptions. GET /api/v1/jobs/search?q=keyword. Assert matching jobs returned, ordered by relevance.
14. **test_contractor_sees_own_jobs_only** — Create jobs for contractor A and B. GET /api/v1/jobs/contractor/mine as contractor A. Assert only A's jobs returned.
15. **test_full_lifecycle_flow** — E2E: create job -> schedule -> in_progress -> complete -> invoiced. Assert all transitions succeed and status_history has 5 entries.
