Create **mobile/lib/features/jobs/presentation/screens/request_review_screen.dart**:
- `class RequestReviewScreen extends ConsumerWidget` -- Admin review queue for incoming job requests
- Watches pendingRequestsNotifier for reactive updates
- Each request card shows: description, client name (or submitted_name for web form submissions), trade type, urgency badge (normal=default, urgent=red), preferred dates, budget range, photo thumbnails (if any)
- Three action buttons per request:
  1. **Accept** (green): Shows confirmation dialog. On confirm, calls `DioClient.post('/api/v1/jobs/requests/{id}/review', data: {'action': 'accepted'})` to trigger server-side job creation via RequestService.review_request. On success (response contains the created Job data), navigate to the job wizard screen with data pre-populated from the response (description, trade_type, client_id, notes) so admin can add contractor/scheduling in Steps 3-4. On failure, show error snackbar with the API error message. Do NOT create the job client-side -- the backend handles job creation and request status update atomically.
  2. **Decline** (red): Shows dialog with dropdown of preset decline reasons (e.g., "Outside service area", "Fully booked", "Service not offered", "Other") + optional message TextField. Calls `DioClient.post('/api/v1/jobs/requests/{id}/review', data: {'action': 'declined', 'decline_reason': reason, 'decline_message': message})`. On success, remove from pending list. On failure, show error snackbar.
  3. **Request Info** (orange): Shows dialog with message TextField. Calls `DioClient.post('/api/v1/jobs/requests/{id}/review', data: {'action': 'info_requested'})`. On success, update request status. On failure, show error snackbar.
- Sorted by created_at ASC (oldest first = review priority)
- Badge on navigation tab showing pending count
- Empty state: "No pending requests" with checkmark icon
