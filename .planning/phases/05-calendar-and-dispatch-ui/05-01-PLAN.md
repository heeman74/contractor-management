    **Create `mobile/lib/features/schedule/data/booking_dao.dart`:**
    Follow the exact pattern from `job_dao.dart`. @DriftAccessor(tables: [Bookings, Jobs, SyncQueue]). Methods:
    - `watchBookingsByContractorAndDate(String contractorId, DateTime date)` — Stream<List<BookingEntity>> filtering by contractorId, date range (dayStart to dayStart+1day), deletedAt.isNull(). Convert rows to BookingEntity.
    - `watchBookingsByCompanyAndDateRange(String companyId, DateTime start, DateTime end)` — Stream<List<BookingEntity>> for calendar views spanning multiple days.
    - `watchUnscheduledJobs(String companyId, DateTime date)` — Stream<List<Job>> returning jobs that have NO booking for the given date. Implementation: SELECT from Jobs LEFT JOIN Bookings ON jobs.id = bookings.jobId AND bookings.deletedAt IS NULL AND bookings.timeRangeStart >= dayStart AND bookings.timeRangeStart < dayStart+1day, WHERE bookings.id IS NULL AND jobs.companyId = companyId AND jobs.deletedAt IS NULL AND jobs.status IN ('quote', 'scheduled'). This Drift query uses `leftOuterJoin` on the Bookings table. The Jobs table must be included in @DriftAccessor tables list for this query.
    - `insertBooking(BookingsCompanion entry)` — transactional dual-write: insert into bookings + insert SyncQueue entry with operation CREATE and JSON payload built manually (not toColumns).
    - `updateBookingTime(String id, DateTime newStart, DateTime newEnd, int currentVersion)` — transactional dual-write: update time range + version bump + SyncQueue UPDATE entry.
    - `softDeleteBooking(String id, int currentVersion)` — transactional dual-write: set deletedAt + version bump + SyncQueue DELETE entry.
    - `upsertBookingFromSync(BookingsCompanion entry)` — for sync pull: insertOnConflictUpdate by id. No sync queue entry (this IS the sync).
    - Helper `_rowToEntity(Booking row)` converting Drift row to BookingEntity.
    - Helper `_buildQueueEntry(...)` following the exact pattern from job_dao.dart.

    Add `DelayReportRequest(BaseModel)` with fields: reason (str, Field(min_length=1)), new_eta (date), version (int). Add docstring explaining this appends a delay entry to status_history and updates scheduled_completion_date.

    **Update `backend/app/features/jobs/service.py`:**
    Add `report_delay(self, job_id: uuid.UUID, data: DelayReportRequest, *, user_id: uuid.UUID) -> Job` method to JobService. Logic:
    1. Fetch job by id (404 if not found)
    2. Check version match (409 if conflict)
    3. Validate job.status in ('scheduled', 'in_progress') — 422 if wrong status
    4. Build delay_entry dict: {"type": "delay", "reason": data.reason, "new_eta": data.new_eta.isoformat(), "timestamp": datetime.now(UTC).isoformat(), "user_id": str(user_id)}
    5. Replace status_history: job.status_history = [*job.status_history, delay_entry] (list replacement, NOT .append())
    6. Update job.scheduled_completion_date = data.new_eta
    7. Increment job.version
    8. flush + refresh + return job
    Import `from datetime import UTC, datetime, date` at top if not already present.

    **Update `backend/app/features/jobs/router.py`:**
    Add `PATCH /jobs/{job_id}/delay` endpoint. CRITICAL: Declare this route BEFORE the `GET /jobs/{job_id}` route to prevent FastAPI route shadowing (see STATE.md: "Route ordering: /jobs/requests* must be declared BEFORE /jobs/{job_id}"). The endpoint accepts DelayReportRequest body, requires get_current_user, delegates to JobService.report_delay(), returns JobResponse. Both contractors (own jobs) and admins can report delays.
